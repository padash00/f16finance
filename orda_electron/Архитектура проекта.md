# Orda Point — Архитектура проекта

> Система управления игровым клубом / точкой продаж. Три клиента на одной БД Supabase.

## Общая схема

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   Веб-портал    │     │ Operator Desktop │     │  Kiosk Desktop  │
│  (Next.js/Web)  │     │   (Electron)     │     │   (Electron)    │
│  ordaops.kz     │     │ desktop/operator │     │ desktop/kiosk   │
└────────┬────────┘     └────────┬─────────┘     └────────┬────────┘
         │                       │                         │
         └───────────────────────┼─────────────────────────┘
                                 │ HTTP + Headers
                    ┌────────────▼─────────────┐
                    │     Next.js API Routes    │
                    │       /api/admin/*        │
                    │       /api/point/*        │
                    │       /api/kiosk/*        │
                    └────────────┬──────────────┘
                                 │
                    ┌────────────▼──────────────┐
                    │         Supabase           │
                    │  PostgreSQL + Auth +       │
                    │  Realtime + Storage        │
                    └───────────────────────────┘
```

**Главное правило**: Supabase доступен ТОЛЬКО через Next.js API. Никакого прямого доступа из Electron.

## Аутентификация

| Клиент | Как передаётся | Где проверяется |
|--------|---------------|-----------------|
| Веб-портал | Supabase Auth (сессионные куки) | `getRequestAccessContext()` |
| Operator Desktop | Заголовок `x-point-device-token` | `requirePointDevice()` |
| Kiosk | `x-device-token` + `x-client-secret` | `/api/kiosk/*` |

### Роли сотрудников
`owner` → `manager` → `other`
SuperAdmin обходит все ограничения (видит все компании).

## Модуль инвентаря — схема БД

```
inventory_items          ← Каталог товаров
  id, name, barcode, unit, sale_price, default_purchase_price

inventory_locations      ← Места хранения (per company)
  id, company_id, location_type (warehouse | point_display)

inventory_balances       ← Текущие остатки
  location_id, item_id → quantity

inventory_receipts       ← Приходы (от поставщика)
inventory_receipt_items

inventory_requests       ← Заявки на перемещение
inventory_request_items    (warehouse → showcase)

inventory_movements      ← История всех движений
```

### Функция одобрения заявки
`inventory_decide_request(request_id, decision, comment)` — Supabase RPC, атомарно:
- Минусует из `inventory_balances` (warehouse)
- Плюсует в `inventory_balances` (showcase)
- Записывает movement

## Паттерн: новый API route

```typescript
// app/api/admin/store/example/route.ts
import { getRequestAccessContext } from '@/lib/server/request-auth'
import { resolveCompanyScope } from '@/lib/server/organizations'
import { createAdminSupabaseClient, hasAdminSupabaseCredentials } from '@/lib/server/supabase'
import { writeAuditLog } from '@/lib/server/audit'

function canView(access: { isSuperAdmin: boolean; staffRole: string }) {
  return access.isSuperAdmin || ['owner', 'manager', 'other'].includes(access.staffRole)
}

export async function GET(request: Request) {
  const access = await getRequestAccessContext(request)
  if ('response' in access) return access.response
  if (!canView(access)) return NextResponse.json({ error: 'forbidden' }, { status: 403 })

  const supabase = hasAdminSupabaseCredentials()
    ? createAdminSupabaseClient()
    : access.supabase

  const companyScope = await resolveCompanyScope({
    activeOrganizationId: access.activeOrganization?.id || null,
    isSuperAdmin: access.isSuperAdmin,
  })

  const query = supabase.from('table').select('...')
  // ВАЖНО: null = superadmin = не фильтровать
  if (companyScope.allowedCompanyIds) {
    query.in('company_id', companyScope.allowedCompanyIds)
  }
}
```
