# Отчет по работам: разделение Склад/Витрина

Дата: 2026-04-22

## Что уже сделано

### 1) Миграция физической модели

- Добавлена миграция `supabase/migrations/20260422_inventory_physical_showcase.sql`.
- В миграции реализовано:
  - заполнение `point_display` из старой виртуальной формулы (`warehouse - backroom`);
  - переприсвоение `warehouse` на физический остаток подсобки;
  - перенос ссылок в истории (`inventory_movements`, `inventory_receipts`, `inventory_writeoffs`, `inventory_stocktakes`) с `backroom/catalog` на `warehouse`;
  - удаление deprecated-локаций `backroom` и `catalog`;
  - сужение `CHECK` по `inventory_locations.location_type` до `('warehouse', 'point_display')`.
- Исправлен SQL-баг `42P01` в шаге обновления `warehouse` (переписан проблемный `UPDATE ... FROM` на коррелированный подзапрос).

### 2) POS и возвраты переведены на физическую витрину

- Обновлены SQL-функции в миграциях:
  - `supabase/migrations/20260420_pos_sale_catalog_decrement.sql`
  - `supabase/migrations/20260420_point_sale_catalog_decrement.sql`
  - `supabase/migrations/20260420_point_return_catalog_increment.sql`
- Логика изменена:
  - продажи уменьшают `point_display` напрямую;
  - возвраты увеличивают `point_display` напрямую.
- Обновлены API:
  - `app/api/pos/sale/route.ts` — проверка/списание с физической витрины;
  - `app/api/pos/return/route.ts` — возврат на физическую витрину.

### 3) Серверные репозитории инвентаря

- Файл: `lib/server/repositories/inventory.ts`
- Изменения:
  - `fetchStoreReceipts`, `fetchStoreWriteoffs`, `fetchStoreRevisions` используют `warehouse + point_display` вместо `catalog`;
  - синхронизация `point_products` теперь ориентируется на активные `point_display` (а не `catalog`).

### 4) Inventory API

- `app/api/admin/inventory/catalog/route.ts`:
  - `total_balance` теперь считается как `warehouse + point_display`;
  - `resetAllBalances` ограничен типами `warehouse/point_display`.
- `app/api/admin/inventory/route.ts`:
  - добавлена поддержка `location_type` + `company_id` для действий:
    - `createReceipt`
    - `createWriteoff`
    - `createStocktake`
  - сервер резолвит `location_id` по `(company_id, location_type)`; старый формат с `location_id` также поддерживается.

### 5) Store API

- `app/api/admin/store/showcase/route.ts`:
  - GET читает физический `point_display` (без виртуальной формулы);
  - `returnToWarehouse` делает реальный перенос `point_display -> warehouse`;
  - cleanup `ensureCompanyLocation` без `catalog`.
- `app/api/admin/store/warehouse/route.ts`:
  - полностью убран runtime-зависимость от `catalog`;
  - GET агрегирует физические остатки `warehouse + point_display`;
  - `addStock` работает с `warehouse`;
  - `setWarehouse` устанавливает физический остаток подсобки;
  - `previewBackroomUpload`/`applyBackroomUpload` обновляют физическую подсобку;
  - `deleteStock` удаляет из `warehouse + point_display`.

### 6) UI (store pages)

- Обновлены типы и подписи локаций:
  - `app/(main)/store/receipts/page.tsx`
  - `app/(main)/store/writeoffs/page.tsx`
  - `app/(main)/store/revisions/page.tsx`
  - `app/(main)/store/movements/page.tsx`
  - `app/(main)/store/showcase/page.tsx`
  - `app/(main)/store/warehouse/page.tsx`
- В `movements` добавлен фильтр "Место":
  - `Все места`
  - `В подсобке`
  - `На витрине`
  - `Между подсобкой и витриной`

### 7) Analytics и ABC

- `app/(main)/store/analytics/page.tsx`:
  - добавлены вкладки:
    - `Витрина (продажи)`
    - `Склад (запасы)`
  - складская вкладка показывает отдельные метрики по запасам.
- `app/api/admin/inventory/abc/route.ts`:
  - добавлен режим `mode=sales|stock`;
  - `sales` — ABC по продажам;
  - `stock` — ABC по стоимости складских остатков (`warehouse`);
  - учтены ограничения доступа по компаниям.
- `app/(main)/store/abc/page.tsx`:
  - сделана отдельная страница с вкладками под оба режима ABC.
